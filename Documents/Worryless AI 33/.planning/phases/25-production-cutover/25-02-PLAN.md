The user must complete these steps in order (Railway dashboard or CLI). The order matters because VITE_ vars are baked at Docker build time.

**Step 1: Assign domains to backend services first**
- Railway Dashboard > API Server service > Settings > Networking > Generate Domain
  - Note the assigned domain (e.g., `api-server-production-xxxx.up.railway.app`)
- Railway Dashboard > LangGraph Server service > Settings > Networking > Generate Domain
  - Note the assigned domain (e.g., `langgraph-server-production-xxxx.up.railway.app`)

**Step 2: Set Frontend build-time vars with the backend domains**
- Railway Dashboard > Frontend service > Variables, set:
  - `VITE_API_URL` = `https://<api-server-domain>.up.railway.app`
  - `VITE_LANGGRAPH_URL` = `https://<langgraph-server-domain>.up.railway.app`
  - `VITE_LOGTO_ENDPOINT` = `https://<logto-domain>.up.railway.app` (should already be set)
  - `VITE_LOGTO_APP_ID` = (Logto SPA app ID, should already be set)
  - `VITE_LOGTO_API_RESOURCE` = (Logto API resource identifier, should already be set)
  - `VITE_VAPID_PUBLIC_KEY` = (VAPID public key, should already be set)
- Trigger a new deploy of the Frontend service (to rebuild Docker with new VITE_ ARGs)

**Step 3: Assign domain to Frontend**
- Railway Dashboard > Frontend service > Settings > Networking > Generate Domain
  - Note the assigned domain (e.g., `frontend-production-xxxx.up.railway.app`)

**Step 4: Set CORS_ORIGIN on API Server**
- Railway Dashboard > API Server service > Variables, set:
  - `CORS_ORIGIN` = `https://<frontend-domain>.up.railway.app`
- Redeploy API Server (setting a variable should auto-trigger redeploy)

**Step 5: Register Logto redirect URI for production**
- Logto Admin Console > Applications > (your SPA app) > Redirect URIs
  - Add: `https://<frontend-domain>.up.railway.app/callback`
- Logto Admin Console > Applications > (your SPA app) > Post Sign-out Redirect URIs
  - Add: `https://<frontend-domain>.up.railway.app`

**Step 6: (If using Google OAuth for Personal Assistant) Update Google Cloud Console**
- Google Cloud Console > APIs & Services > Credentials > OAuth Client
  - Add authorized redirect URI: `https://<logto-domain>.up.railway.app/callback/google-universal`

**Step 7: Verify get_due_cadence_agents() SQL function exists**
- Connect to Railway Postgres and run: `SELECT proname FROM pg_proc WHERE proname = 'get_due_cadence_agents';`
- If missing, apply the standalone migration: `psql $DATABASE_URL -f worrylesssuperagent/supabase/migrations/20260321000001_create_agent_tables.sql` (or the dedicated cadence migration file)

User performs a browser-based smoke test across 5 tiers. Tiers 1-3 are required for phase completion. Tiers 4-5 are optional (depend on time and API key availability). Use an incognito/private browser window to simulate a first-time user.

**Tier 1: Service Availability**
1. Open `https://<frontend-domain>.up.railway.app/` -- should render the landing page or Logto login redirect (not blank, not 404)
2. Open `https://<api-server-domain>.up.railway.app/health` -- should show `{"status":"ok","timestamp":"..."}`
3. Open `https://<langgraph-domain>.up.railway.app/health` -- should show `{"status":"ok","checkpointer":"connected","store":"connected"}`

**Tier 2: Auth Flow**
4. Click Sign In on the frontend -- should redirect to Logto sign-in page (not a blank page, not `redirect_uri_mismatch` error)
5. Create a new account with email/password -- should land on the onboarding flow
6. Complete all 11 onboarding steps -- should reach the Dashboard with the Chief of Staff agent visible

**Tier 3: Core Agent Interaction**
7. Open Chief of Staff chat -- type "What can you help me with?" -- response should stream token-by-token (not appear all at once after a long delay)
8. Verify agent team activation -- at least one specialist agent visible in the team view

**Tier 4: Background Systems (optional -- may take up to 10 minutes)**
9. Enable cadence on an agent -- wait 5-10 minutes -- check if an insight appears in the dashboard without manual prompt
10. Toggle push notification opt-in -- browser should prompt for push permission

**Tier 5: External Integrations (optional -- requires valid API keys)**
11. Ask the Marketer agent to create a brand image -- image should render in chat (base64 data URI, not broken `<img>` tag)
12. Trigger a test email from Settings -- email should arrive in inbox via Resend

**Troubleshooting guide if issues occur:**
- Blank page on frontend: Check browser devtools console for `VITE_API_URL is not set` -- means Frontend needs rebuild with correct env vars
- CORS error on API calls: Check that `CORS_ORIGIN` on API Server matches the exact frontend domain (including `https://`)
- `redirect_uri_mismatch` on Logto: The production callback URL is not registered in Logto admin console
- Chat not streaming: Check that `X-Accel-Buffering: no` header is present in LangGraph proxy responses (Network tab > Response Headers)
- "GEMINI_API_KEY environment variable is required" error: GEMINI_API_KEY not set on LangGraph Server service in Railway
