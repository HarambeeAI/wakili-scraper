    LLM parsing behavior (for test stubs → real tests):
    - extractJson("```json\n{\"severity\":\"ok\",\"finding\":\"\"}\n```") → '{"severity":"ok","finding":""}'
    - parseSeverity('{"severity":"urgent","finding":"Invoice overdue"}') → { severity: "urgent", finding: "Invoice overdue" }
    - parseSeverity("not valid json") → { severity: "ok", finding: "" }
    - parseSeverity('{"finding":"something"}') (missing severity) → { severity: "ok", finding: "" }
    - parseSeverity('{"severity":"unknown","finding":"x"}') (invalid value) → { severity: "ok", finding: "" }

    Runner behavior:
    - Dequeues up to 5 messages with 30s visibility timeout
    - For each message: fetches HEARTBEAT.md workspace content for that agentTypeId + userId
    - Fetches last 5 tasks from tasks table for context (or empty array if none)
    - Calls LLM with system prompt containing sanitized HEARTBEAT.md + task context
    - Parses response with extractJson + parseSeverity (fail-safe to ok)
    - severity "ok": returns immediately (no DB writes)
    - severity "urgent"/"headsup"/"digest": inserts agent_heartbeat_log with outcome='surfaced'
    - severity "urgent"/"headsup": additionally inserts notifications row
    - severity "urgent": additionally calls sendUrgentEmail in independent try/catch
    - After successful processing: deletes message from pgmq
    - On any processing error: does NOT delete message (becomes re-visible after 30s)
  </behavior>
