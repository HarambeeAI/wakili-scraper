  <read_first>
    - .planning/phases/19-infrastructure-provisioning/19-RESEARCH.md (Pattern 2: Logto dedicated database, Pattern 3: reference variables, Anti-patterns: domain before ADMIN_DISABLE_LOCALHOST)

  </read_first>
  <files></files>
  <action>
    Deploy Logto using Railway MCP tools:

    1. First, try deploying Logto using the Railway template:
       Call `mcp__railway-mcp-server__deploy-template` with template URL `https://railway.com/deploy/logto`.

       If the template is not available or deploys its own Postgres (which we do NOT want), instead use:
       Call `mcp__railway-mcp-server__deploy` with Docker image `logto/logto:latest`.

    2. After the Logto service is created, call `mcp__railway-mcp-server__list-services` to confirm the Logto service name and note it for reference variables.

    3. CRITICAL ORDER -- Generate domains BEFORE setting ADMIN_DISABLE_LOCALHOST:

       a. Call `mcp__railway-mcp-server__generate-domain` for the Logto service on port 3001 (auth endpoint).
       b. Call `mcp__railway-mcp-server__generate-domain` for the Logto service on port 3002 (admin console).

       Note the generated domain URLs. If Railway only allows one domain per service, use port 3001 as the primary domain. The admin console may need to be accessed differently (document what happens).

    4. Set Logto service variables using `mcp__railway-mcp-server__set-variables` on the Logto service with these exact variables (use the Postgres service name from 19-01 SUMMARY for reference variables):

       ```
       DB_URL=postgresql://${{Postgres.PGUSER}}:${{Postgres.PGPASSWORD}}@${{Postgres.RAILWAY_PRIVATE_DOMAIN}}:5432/logto
       ENDPOINT=https://<auth-domain-from-step-3a>
       ADMIN_ENDPOINT=https://<admin-domain-from-step-3b>
       TRUST_PROXY_HEADER=1
       PORT=3001
       ADMIN_PORT=3002
       ```

       IMPORTANT: If the Postgres service has a different name than "Postgres" (check list-services output from 19-01), adjust the reference variable prefix accordingly (e.g., `${{pgvector-18-trixie.PGUSER}}`).

       Do NOT set `ADMIN_DISABLE_LOCALHOST=1` yet -- wait until admin console is confirmed working.

    5. If the Logto service was deployed via `deploy` (not template), it may need a redeploy to pick up the new variables. Call `mcp__railway-mcp-server__deploy` to trigger a redeploy.

    6. Wait ~60 seconds for Logto to start and seed its database, then call `mcp__railway-mcp-server__get-logs` on the Logto service. Look for:
       - "seeding complete" or "Logto is running" or "server started" messages
       - No "database does not exist" errors (if this appears, the `logto` database was not created in 19-01 Task 2)
       - Successful connection to `postgres.railway.internal:5432/logto`

    7. After confirming Logto is running, set `ADMIN_DISABLE_LOCALHOST=1` via `mcp__railway-mcp-server__set-variables` on the Logto service:
       ```
       ADMIN_DISABLE_LOCALHOST=1
       ```
  </action>
