Create `supabase/functions/proactive-runner/index.ts` — a Deno Edge Function that:

1. **Dequeues** up to 5 messages from `heartbeat_jobs` queue using the exact pgmq read pattern from heartbeat-runner (schema("pgmq_public").rpc("read", { queue_name: "heartbeat_jobs", sleep_seconds: 30, n: 5 }))

2. **For each message**, extracts: `user_id`, `agent_type_id`, `cadence_tier` (default to "daily" if missing — backward compatibility with messages enqueued by the old dispatcher)

3. **Feature flag gate**: Checks `profiles.use_langgraph` for the user. If `false` or missing, SKIP the message (do NOT delete — let it be picked up by the old heartbeat-runner after visibility timeout expires). Log: `[proactive-runner] Skipping non-LangGraph user ${userId}`.

4. **Construct heartbeat prompt**: Import the prompt strings inline (cannot import from langgraph-server in Deno). Define a `HEARTBEAT_PROMPTS` constant directly in the file (same content as langgraph-server/src/cadence/heartbeat-prompts.ts but duplicated for Deno compatibility). Use `cadence_tier` to select the correct prompt. Fallback to daily if tier not found.

   WAIT — actually, the proactive-runner calls `/invoke` on the LangGraph server. The server receives the prompt as the `message` field. The LangGraph server's agent graph already has tool dispatch nodes that classify the message. So the heartbeat prompt IS the message passed to /invoke. The proactive-runner does NOT need the prompt classification logic — it just needs the prompt STRINGS.

   Define the prompts inline (copy from 16-01 heartbeat-prompts.ts since we cannot cross-import between Deno and Node):

   ```typescript
   const HEARTBEAT_PROMPTS: Record<string, Record<string, string>> = { ... };
   ```

   Alternatively, to avoid duplication: the proactive-runner can use a simpler approach — pass a generic message like `"Run your ${cadenceTier} heartbeat checklist"` and let the LangGraph server's agent graph handle the prompt construction. BUT this won't trigger the right regex classification. The regex needs specific keywords.

   DECISION: Duplicate the prompt strings in the Edge Function. This is the pragmatic choice (Deno cannot import Node modules). Keep them short — just the prompt text, ~10 lines per agent.

5. **Construct proactive thread ID**: `proactive:${agentTypeId}:${userId}` — deterministic, persistent across runs.

6. **Call LangGraph /invoke**:
   ```typescript
   const LANGGRAPH_SERVER_URL = Deno.env.get("LANGGRAPH_SERVER_URL");
   // Fail early if not set
   if (!LANGGRAPH_SERVER_URL) throw new Error("LANGGRAPH_SERVER_URL not configured");

   const response = await fetch(`${LANGGRAPH_SERVER_URL}/invoke`, {
     method: "POST",
     headers: { "Content-Type": "application/json" },
     body: JSON.stringify({
       message: heartbeatPrompt,
       user_id: userId,
       thread_id: `proactive:${agentTypeId}:${userId}`,
       agent_type: agentTypeId,
     }),
   });
   ```

7. **Parse response**: The /invoke endpoint returns `{ response: string, ... }`. The `response` is the last LLM message. Parse it for severity using a simplified severity extraction (look for JSON with severity field, or if the response mentions "urgent"/"headsup"/"digest" keywords, map accordingly). Use `extractJson` and `parseSeverity` from `../_shared/heartbeatParser.ts` (these are Deno-compatible shared imports).

8. **Write to agent_heartbeat_log**: Same INSERT pattern as heartbeat-runner. Only write if severity !== "ok".

9. **Write to notifications**: Same INSERT pattern as heartbeat-runner for "urgent" and "headsup" severities.

10. **Advance next-tier timestamp**: After successful processing, advance the relevant next_*_heartbeat_at column:
    - daily: `next_heartbeat_at = now() + interval_hours` (same as current dispatcher)
    - weekly: `next_weekly_heartbeat_at = now() + 7 days`
    - monthly: `next_monthly_heartbeat_at = now() + 1 month`
    - quarterly: `next_quarterly_heartbeat_at = now() + 3 months`

    ```typescript
    const advanceColumn: Record<string, string> = {
      daily: "next_heartbeat_at",
      weekly: "next_weekly_heartbeat_at",
      monthly: "next_monthly_heartbeat_at",
      quarterly: "next_quarterly_heartbeat_at",
    };
    const advanceInterval: Record<string, number> = {
      daily: agent.heartbeat_interval_hours * 60 * 60 * 1000,
      weekly: 7 * 24 * 60 * 60 * 1000,
      monthly: 30 * 24 * 60 * 60 * 1000,
      quarterly: 90 * 24 * 60 * 60 * 1000,
    };
    ```

    NOTE: The proactive-runner advances the timestamp, NOT the dispatcher. The dispatcher only enqueues. This prevents Pitfall 6 (column not advanced after enqueue).

    WAIT — the current dispatcher advances next_heartbeat_at. For consistency, keep the dispatcher advancing daily, and have the proactive-runner advance weekly/monthly/quarterly ONLY. The dispatcher already advances daily's next_heartbeat_at. For weekly/monthly/quarterly, the dispatcher should advance them too (same pattern). Update the dispatcher in Task 2.

11. **Delete message from queue** only after successful processing (same as heartbeat-runner).

12. **Token budget bypass**: Pass `is_proactive: true` in the /invoke body. The LangGraph server's /invoke does NOT currently read this field — this is a forward-compatible flag. For now, proactive runs DO count against budget (acceptable per research: prompts are short). Future: the server can read this flag and skip budget checks.

13. **Error handling**: Wrap each message processing in try/catch. Failed messages are NOT deleted from queue (re-appear after visibility timeout). Log errors with `[proactive-runner]` prefix.

14. **Imports**: Use Deno-compatible imports:
